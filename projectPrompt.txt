Project Goal:

Rebuild a Square Payment Gateway Adapter for FOSSBilling from scratch using a clean, stable, and maintainable architecture.

The adapter must:
- Fully support one-time invoice payments
- Fully support recurring subscriptions using Square
- Integrate with FOSSBilling’s invoice system without modifying core templates
- Avoid fragile dependencies on banklink routes or default checkout form submission
- Use Square Web Payments SDK for secure card tokenization
- Use backend API processing for payment execution

The system must be separated into clear layers:
1. Payment Adapter (core payment logic)
2. Frontend Checkout UI (Square integration)
3. Extension Module (subscription management and admin tools)

Key constraints:
- No reliance on banklink POST submissions
- No nested or conflicting forms inside FOSSBilling checkout
- No dependency on PayPal-style redirect logic
- Must be compatible with FOSSBilling’s API and transaction lifecycle

Primary design approach:
- Treat Square as a direct API-based gateway (NOT redirect-based)
- Tokenize card data in browser
- Send secure token to backend
- Process payment immediately via adapter

The adapter must remain self-contained and usable independently of the extension module.
Architecture Contract:

The Square payment adapter must follow a strict and minimal contract aligned with FOSSBilling’s payment system.

The adapter must be implemented as an API-style payment gateway, not a redirect-style gateway.

Core Responsibilities:

1. getHtml()
   - Render the payment UI only
   - Output minimal HTML required for Square card entry
   - Inject configuration into the frontend via a JavaScript-safe object
   - Load Square Web Payments SDK
   - MUST NOT:
       - Submit forms
       - Redirect to banklink
       - Trigger checkout submission
       - Throw exceptions that break page rendering

2. Frontend Checkout Logic (JS)
   - Initialize Square SDK
   - Render card input fields
   - Handle user interaction (Pay button click)
   - Tokenize card data securely
   - Send payment token to backend via HTTP request (AJAX/fetch)
   - MUST:
       - Prevent default page submission behavior
       - Never rely on FOSSBilling checkout form submission
       - Never post to /invoice/banklink/

3. processTransaction()
   - Accept a secure payment token from frontend
   - Validate required fields (invoice_id, gateway_id, token)
   - Call Square API to process payment
   - Update FOSSBilling:
       - Record transaction
       - Mark invoice paid
       - Handle subscription setup if applicable

4. Separation of Concerns
   - Adapter handles payments only
   - Extension module handles:
       - Plan/product mapping
       - Subscription configuration
       - Admin UI
       - Data export/import
   - Frontend JS handles only tokenization + request dispatch

5. Error Handling Rules
   - getHtml() must NEVER throw fatal exceptions
   - All runtime errors should be handled:
       - safely in JS (user feedback)
       - safely in backend (return structured errors)
   - No errors should break Twig rendering or invoice views

6. Data Flow

   Browser:
       User clicks "Pay"
       → Tokenize card via Square SDK
       → Send token to backend API

   Backend:
       → Receive token
       → Process payment via Square API
       → Update invoice/transaction state
       → Return success or failure

7. Strict Prohibitions
   The implementation must NOT:
   - Use FOSSBilling banklink POST flow
   - Use nested or injected forms
   - Submit to current page URL
   - Copy PayPal adapter behavior
   - Depend on checkout DOM structure outside its own container

Implementation Specification — getHtml() Rendering

The getHtml() method must ONLY handle rendering of the payment UI and configuration delivery.

It must follow these strict requirements:

--------------------------------------------------

1. Input Handling

The method receives:
  - $invoice_id
  - $subscription flag

It must:
  - Safely load the invoice using the database layer
  - Convert the invoice to an array if needed
  - NEVER assume fields exist (use safe defaults)

Required extracted values:
  - invoiceId (integer)
  - amount (formatted string, "0.00" format)
  - currency (uppercase string, default "USD")

No field access may cause PHP warnings or exceptions.

--------------------------------------------------

2. Gateway Identification

The method must:
  - Retrieve gateway_id from pay_gateway table
  - Use safe fallback if not found (set to 0, log warning)

It must NEVER:
  - throw fatal exceptions
  - break rendering if gateway is misconfigured

--------------------------------------------------

3. Configuration Object

A JavaScript-safe configuration object (squareConfig) must be created.

Required fields:
  - applicationId
  - locationId
  - invoiceId
  - gatewayId
  - processUrl
  - amount
  - currency
  - environment
  - isSubscription

All values must be:
  - explicitly cast to correct type
  - never null (use safe defaults)

--------------------------------------------------

4. HTML Output Structure

The method must render ONLY:

<div id="square-checkout-root">
  <div id="square-card-container"></div>
  <button id="square-pay-button">Pay</button>
  <div id="square-payment-status"></div>

Rules:
  - No <form> elements
  - No nested forms
  - No hidden inputs
  - No dependency on existing checkout form
  - No inline PHP variables inside HTML attributes unless escaped

--------------------------------------------------

5. Script Injection

The method must inject:

A. squareConfig:
<script>

B. Square SDK:
<script src="https://web.squarecdn.com/v1/square.js"></script>

C. Inline checkout JS:

--------------------------------------------------

6. Output Safety Rules

getHtml() MUST:
  - NEVER throw exceptions
  - NEVER return null
  - ALWAYS return valid HTML string
  - NEVER modify global state
  - NEVER interfere with Twig rendering

--------------------------------------------------

7. Strict Prohibitions

The method must NOT:
  - Submit forms
  - Redirect pages
  - Call processTransaction()

  - Trigger payment processing
  - Depend on external JS files (must be inline)

--------------------------------------------------

8. Final Responsibility

getHtml() is strictly:

→ "Render UI + Provide Config"

→ "Control Payment Execution"

Implementation Specification — Frontend Checkout Logic (JavaScript)

The frontend script is responsible for handling user interaction, card tokenization, and communication with the backend.

It must operate completely independently from FOSSBilling’s default checkout behavior.

--------------------------------------------------

1. Initialization

When the page loads:

  - Locate #square-checkout-root
  - Read configuration from window.squareConfig
  - Validate required fields:
      - applicationId
      - locationId
      - invoiceId
      - gatewayId
      - processUrl

If any required field is missing:
  → Log error in console
  → Display user-friendly message in UI
  → DO NOT proceed further

--------------------------------------------------

2. Square SDK Setup

The script must:

  - Initialize Square using:
        Square.payments(applicationId, locationId)

  - Create card instance:
        payments.card()

  - Attach card to:
        #square-card-container

If initialization fails:
  → Display "Payment system unavailable"
  → Disable payment button

--------------------------------------------------

3. Pay Button Behavior

On click of #square-pay-button:

  - MUST call:
        event.preventDefault()
        event.stopPropagation()

  - MUST disable button to prevent double payments

  - MUST NOT:
        - submit any form
        - trigger page navigation
        - interact with external DOM elements

--------------------------------------------------

4. Tokenization Flow

On click:

  → Call:
        card.tokenize()

If result.status !== "OK":

  - Extract error messages
  - Display errors in #square-payment-status
  - Re-enable button
  - STOP execution

If successful:

  - Extract:
        result.token

--------------------------------------------------

5. API Request

Send token to backend using fetch:

  Method:
        POST

  URL:
        window.squareConfig.processUrl

  Body:
        {
            invoice_id: <invoiceId>,
            gateway_id: <gatewayId>,
            source_token: <token>
        }

  Headers:
        Content-Type: application/json

  MUST:
        - include credentials: same-origin
        - handle non-JSON responses safely

--------------------------------------------------

6. Response Handling

On response:

  IF success:
      - Show "Payment successful"
      - Reload page to update invoice state

  IF error:
      - Extract error message safely
      - Display error in UI
      - Re-enable button

  MUST:
      - NOT throw unhandled JS errors
      - NOT break UI
      - ALWAYS give user feedback

--------------------------------------------------

7. Error Handling Rules

The script must:

  - Catch ALL async errors
  - Log detailed errors to console
  - Show simplified messages to user

Example:

  console.error(...)
  user sees:
      "Payment failed. Please try again."

--------------------------------------------------

8. DOM Scope Rules

The script must ONLY interact with:

  - #square-checkout-root
  - Its child elements

It must NOT:

  - Modify FOSSBilling checkout form
  - Query global form elements
  - Override site-wide JS behavior

--------------------------------------------------

9. Strict Prohibitions

The script must NOT:

  - Call form.submit()
  - Redirect browser manually
  - Post to /invoice/banklink/
  - Use jQuery-specific APIs
  - Depend on bb.post or FOSSBilling internal JS helpers

--------------------------------------------------

10. Final Responsibility

The frontend is strictly:

  → "Tokenize card + send request"

NOT:

  → "Process payment"
  → "Control invoice state"

Implementation Specification — processTransaction()

The processTransaction() method is the ONLY place where actual payment processing occurs.

It must safely receive a token, call Square API, and update the invoice + transaction state.

--------------------------------------------------

1. Method Responsibility

This method must:

  - Receive payment request from frontend/API
  - Validate incoming data
  - Execute Square payment
  - Record transaction
  - Mark invoice as paid
  - Handle subscription creation (if enabled)

It must NOT:

  - Render output
  - Echo/print anything
  - Break application flow

--------------------------------------------------

2. Expected Inputs

The method receives:

  - $id (transaction ID)
  - $data (request payload)
  - $gateway_id

From $data, extract:

  - invoice_id
  - source_token (Square token)

Validation rules:

  - invoice_id must exist and be a valid integer
  - source_token must be a non-empty string
  - gateway_id must match current gateway

If any value is invalid:
  → throw controlled exception (NOT fatal crash)

--------------------------------------------------

3. Load Required Models

The method must load:

  - Transaction model (using $id)
  - Invoice model (using invoice_id)
  - Gateway configuration
  - Client information (via invoice)

If anything fails:
  → throw safe exception

--------------------------------------------------

4. Prepare Square Payment Request

Construct payment request using:

  - source_token
  - amount (converted to smallest currency unit, e.g. cents)
  - currency
  - idempotency_key (unique per transaction)

Example:

  amount_money:
      amount = invoice amount * 100
      currency = invoice currency

Important:
  - Amount must match invoice total exactly
  - Never trust frontend amount

--------------------------------------------------

5. Call Square API

Use Square SDK or HTTP client:

  - Endpoint: create payment
  - Pass token and amount

Handle response carefully:

  IF success:
      → proceed to update system

  IF failure:
      → extract error message
      → throw controlled exception

--------------------------------------------------

6. Transaction Recording

Update FOSSBilling transaction:

  - txn_id → Square payment ID
  - txn_status → "completed" or equivalent
  - amount → invoice total
  - currency → invoice currency
  - status → PROCESSED
  - updated_at → current timestamp

Must:
  - store result in DB
  - ensure consistency

--------------------------------------------------

7. Mark Invoice as Paid

Use invoice service:

  - approve invoice if not approved
  - apply payment using credits system

Example logic:

  IF invoice is not approved:
      → approve it

  THEN:
      → mark as paid

--------------------------------------------------

8. Subscription Handling (if applicable)

If subscription flag is active:

  - Create subscription record in FOSSBilling
  - Store:
        - Square subscription ID
        - billing period
        - amount
        - currency

  - Link subscription to invoice

This step may call helper methods or extension services.

--------------------------------------------------

9. Error Handling

All errors must:

  - be caught at appropriate level
  - produce controlled exceptions
  - NOT corrupt transaction data
  - NOT leave invoice in inconsistent state

Example rule:

  IF Square payment fails:
      → DO NOT mark invoice as paid
      → return failure response

--------------------------------------------------

10. Idempotency Protection

The system must:

  - Avoid duplicate charges
  - Use unique idempotency_key for each transaction attempt

If duplicate detected:
  → safely ignore duplicate processing

--------------------------------------------------

11. Return Behavior

The method should:

  - Return success status OR
  - Throw controlled exception

It must NOT:

  - redirect user
  - generate HTML
  - manipulate frontend

--------------------------------------------------

12. Security Requirements

The method MUST:

  - NEVER accept raw card data
  - ONLY accept tokenized input
  - NEVER log sensitive payment details
  - Validate all inputs before use

--------------------------------------------------

13. Strict Prohibitions

The method must NOT:

  - use banklink routes
  - rely on GET/POST globals directly
  - assume valid frontend behavior
  - trust any client-supplied values except token

--------------------------------------------------

14. Final Responsibility

processTransaction() is strictly:

  → "Execute payment + update system state"

NOT:

  → "Render UI"
  → "Handle checkout flow"
  → "Control frontend"  
  

Implementation Specification — API Integration Layer

The API integration layer defines how the frontend securely hands off a tokenized payment request to the backend so that processTransaction() can execute cleanly inside FOSSBilling.

--------------------------------------------------

1. Purpose

This layer exists to connect:

  Frontend:
      tokenized Square payment request

  Backend:
      adapter processTransaction()

It must provide a safe, minimal, and explicit handoff.

--------------------------------------------------

2. Required Behavior

The frontend must send a single payment request containing ONLY:

  - invoice_id
  - gateway_id
  - source_token

These values must be sent to a backend endpoint that is explicitly designed to initiate payment processing.

The request must:
  - be POST
  - use JSON payload
  - include same-origin credentials
  - expect JSON response

--------------------------------------------------

3. Endpoint Contract

The API endpoint used by the adapter must support:

  Request payload:
      {
          invoice_id: integer,
          gateway_id: integer,
          source_token: string
      }

  Response payload:
      success:
          {
              success: true,
              transaction_id: ...,
              status: ...
          }

      failure:
          {
              success: false,
              error: ...
          }

The adapter/frontend contract must not assume any undocumented response shape.
It must safely handle unexpected or malformed responses.

--------------------------------------------------

4. Gateway Routing Requirement

The endpoint must route the request to the correct payment gateway using:

  gateway_id

The request must never rely on:
  - gateway title
  - gateway name string
  - inferred gateway type

Only the actual gateway ID should be used for routing.

--------------------------------------------------

5. Transaction Triggering

The endpoint must result in the backend eventually invoking:

  processTransaction()

The frontend must not directly attempt to:
  - mark invoices paid
  - update transactions
  - create subscriptions

Only the backend performs those operations.

--------------------------------------------------

6. Security Rules

The API request must:

  - NEVER include raw card data
  - ONLY include Square token
  - validate invoice ownership / access rules
  - reject incomplete payloads
  - reject invalid gateway IDs
  - reject missing tokens

The API response must not expose:
  - secrets
  - internal stack traces
  - raw Square credentials

--------------------------------------------------

7. Failure Handling

If the endpoint fails:

  Frontend must:
      - show user-friendly payment failure message
      - re-enable the Pay button
      - keep user on current page

  Backend must:
      - return structured error
      - not mark invoice paid
      - not partially create subscriptions
      - not corrupt transaction records

--------------------------------------------------

8. Non-UI Rule

The API endpoint must behave as a pure backend process.

It must NOT:
  - return HTML
  - redirect browser
  - depend on banklink page rendering
  - require checkout form submission

--------------------------------------------------

9. Stability Requirement

If the endpoint is unavailable or misconfigured:

  - payment UI must still render
  - user must receive a clear failure message
  - invoice page must not crash

The API integration layer must fail safely.

--------------------------------------------------

10. Strict Prohibitions

This layer must NOT:

  - use /invoice/banklink/ as a processing endpoint
  - rely on redirect-based payment flow
  - use nested forms
  - depend on default checkout submit handlers
  - assume a PayPal-style external postback lifecycle

--------------------------------------------------

11. Final Responsibility

The API integration layer is strictly:

  → "Deliver tokenized payment request to backend"

NOT:

  → "Execute payment in browser"
  → "Replace transaction logic"
  → "Control invoice rendering"
 

Implementation Specification — Subscription Creation & Recurring Billing Rules

This section defines when and why subscription logic should run inside the rebuilt Square payment adapter.

--------------------------------------------------

1. Purpose

Subscription logic exists to allow the Square payment adapter to support recurring billing in FOSSBilling.

It must handle:

  - identifying recurring invoices
  - creating Square subscriptions when required
  - storing subscription linkage back into FOSSBilling

It must NOT interfere with one-time invoice payments.

--------------------------------------------------

2. Trigger Condition

Subscription logic must run ONLY when the current payment represents a recurring billing scenario.

The adapter must clearly determine:

  - whether the current invoice is subscription-related
  - whether the payment is intended to start a recurring billing agreement
  - whether a one-time charge should be processed instead

If those conditions are not met:
  → the adapter must follow one-time payment logic only

--------------------------------------------------

3. Separation from One-Time Payments

The system must maintain two clean paths:

  Path A:
      One-time payment processing

  Path B:
      Subscription setup and recurring billing processing

These paths must be explicitly separated in code.

The adapter must NOT:
  - guess
  - mix the flows together
  - create subscriptions during one-time invoice payment processing by accident

--------------------------------------------------

4. Subscription Detection Rule

The adapter must use a reliable internal method to determine if the invoice represents a recurring service.

This detection must be based on normalized invoice/order/subscription data and not on fragile UI assumptions.

The detection logic must be:

  - explicit
  - reusable
  - testable
  - isolated in a helper method

--------------------------------------------------

5. First Payment Rule

The first successful subscription-related payment must result in:

  - a valid Square customer
  - a stored reusable card/payment source if required
  - creation of a recurring subscription in Square
  - linkage back to FOSSBilling

This first payment flow must not rely on future callbacks to create the subscription.

--------------------------------------------------

6. Renewal Rule

Renewal invoices must not behave like brand-new subscriptions unless necessary.

The system must distinguish between:

  - initial subscription creation
  - recurring renewal collection
  - failed subscription retry cases

--------------------------------------------------

7. Stability Rule

Subscription logic must fail safely.

If subscription creation fails:

  - invoice must NOT be falsely marked as fully paid
  - transaction state must remain consistent
  - user must receive a clear failure response

--------------------------------------------------

8. Strict Prohibitions

Subscription logic must NOT:

  - run for every invoice by default
  - depend on frontend assumptions
  - depend on PayPal-style recurring flow
  - depend on banklink redirects
  - create duplicate subscriptions

--------------------------------------------------

9. Final Responsibility

This layer is strictly:

  → "Decide whether recurring billing must be created"

NOT:

  → "Render checkout UI"
  → "Own payment tokenization"
  → "Replace transaction processing"
 

Implementation Specification — Subscription Data Model & Required Stored Fields

This section defines what subscription-related data must exist, what must be stored, and how the adapter/extension should treat recurring billing records.

--------------------------------------------------

1. Purpose

The subscription data model exists to give the Square adapter and the Square extension a stable internal structure for recurring billing.

It must support:

  - identifying recurring services
  - mapping Square subscriptions to FOSSBilling records
  - storing billing metadata safely
  - supporting renewals, cancellations, and status updates

--------------------------------------------------

2. Core Subscription Record Requirements

The system must be able to store or derive the following fields for each recurring billing relationship:

  - client_id
  - invoice_id (initial invoice or linked invoice)
  - gateway_id
  - currency
  - amount
  - billing period
  - recurring interval/cycle
  - status
  - external subscription identifier (Square subscription ID)
  - external customer identifier (Square customer ID)
  - external payment source/card identifier if applicable

These values must be treated as authoritative linkage data between FOSSBilling and Square.

--------------------------------------------------

3. Required Internal Meanings

Each field must have a clear meaning:

  client_id
      The FOSSBilling client who owns the recurring billing relationship

  invoice_id
      The invoice associated with the initial subscription creation or current linked billing action

  gateway_id
      The FOSSBilling payment gateway record used for this subscription

  currency
      The billing currency used for the subscription

  amount
      The recurring charge amount expected for each cycle

  billing period / interval
      The recurring cadence used for billing
      Example concepts:
          weekly
          monthly
          quarterly
          yearly

  status
      The current subscription state
      Example concepts:
          active
          pending
          canceled
          failed

  Square subscription ID
      The external recurring billing identifier created in Square

  Square customer ID
      The external customer record tied to the billing agreement

  card/payment source ID
      The reusable stored payment source associated with the recurring billing setup

--------------------------------------------------

4. Mapping Requirement

The data model must support a mapping between:

  FOSSBilling recurring product/billing definition
      ↔
  Square recurring billing entity

This mapping must be explicit and persistent.

It must NOT rely on:
  - fragile display labels
  - guessed plan names
  - assumptions from frontend UI

--------------------------------------------------

5. Product/Plan Association Requirement

For subscription setup, the system must support storage or lookup of:

  - FOSSBilling product identifier
  - selected billing interval
  - matching Square recurring plan/variation identifier

This association must be stable enough to:
  - create subscriptions correctly
  - prevent duplicate or wrong plan selection
  - allow admin management through the extension

--------------------------------------------------

6. Initial Payment Linkage Rule

When a subscription is first created, the data model must link together:

  - client
  - invoice
  - transaction
  - Square customer
  - Square payment source
  - Square subscription ID

The system must be able to reconstruct the recurring billing relationship from persisted records.

--------------------------------------------------

7. Renewal Compatibility Requirement

The data model must support later processing of:

  - renewals
  - retries
  - cancellations
  - status synchronization

This means recurring records must not only store “creation-time” values.
They must also persist enough identifiers to manage the subscription over time.

--------------------------------------------------

8. Persistence Rule

If the extension introduces its own tables or metadata storage, those records must be used only for:

  - mapping
  - administrative support
  - subscription helper data

The core payment adapter must remain able to function even if optional admin metadata is limited, provided the critical subscription identifiers exist.

--------------------------------------------------

9. Validation Requirements

Before using subscription data, the system must validate:

  - client_id exists
  - invoice_id exists
  - gateway_id is valid
  - amount is numeric and positive
  - currency is valid
  - external Square IDs are non-empty when required

If required fields are missing:
  → subscription creation/update must fail safely

--------------------------------------------------

10. Strict Prohibitions

The subscription data model must NOT:

  - depend on frontend state
  - depend on hardcoded plan assumptions
  - overwrite unrelated invoice data
  - assume every recurring invoice is an initial signup
  - treat one-time invoices as recurring records

--------------------------------------------------

11. Final Responsibility

This layer is strictly:

  → "Define what recurring billing data must exist and persist"

NOT:

  → "Create subscriptions in frontend"
  → "Render checkout UI"
  → "Execute payment tokenization"


Implementation Specification — Subscription Creation Flow & Lifecycle Rules

This section defines how recurring billing must be created, linked, maintained, and transitioned over time inside the rebuilt Square payment adapter.

--------------------------------------------------

1. Purpose

The subscription creation flow exists to define the exact lifecycle of a recurring billing relationship from:

  initial signup
      →
  active recurring billing
      →
  renewal / retry / cancellation / failure

The system must treat subscription lifecycle as a backend-controlled process.

--------------------------------------------------

2. Initial Subscription Creation Flow

When a subscription-related payment is being processed for the first time, the backend must execute these steps in order:

  1. validate recurring billing intent
  2. load invoice and client
  3. identify selected recurring plan/mapping
  4. ensure Square customer exists
  5. create or store reusable payment source if required
  6. create Square subscription
  7. store resulting Square subscription identifiers
  8. update FOSSBilling subscription record
  9. finalize transaction/invoice state

Each step must succeed before the next one is considered valid.

--------------------------------------------------

3. Preconditions Before Subscription Creation

Before a Square subscription can be created, the backend must confirm:

  - invoice is valid
  - client is valid
  - recurring billing is intended
  - mapped recurring plan exists
  - gateway is configured
  - tokenized payment source is valid
  - required billing metadata exists

If any precondition fails:
  → subscription creation must stop safely
  → invoice must remain unpaid unless explicitly supported by design
  → no partial recurring state should be recorded as active

--------------------------------------------------

4. Customer Creation Rule

The lifecycle must include a stable Square customer record.

If Square customer does not already exist:
  → create one

If it already exists:
  → reuse it

The adapter must not create duplicate customer records unnecessarily.

--------------------------------------------------

5. Payment Source / Card Storage Rule

If Square recurring billing requires a stored payment method:

  - create/store the reusable payment source
  - link it to the correct Square customer
  - persist the resulting external identifier if needed

The system must clearly distinguish between:

  - one-time source token
  - reusable stored payment source
  - external subscription object

--------------------------------------------------

6. Recurring Plan Resolution Rule

Before creating the subscription, the adapter must resolve which Square recurring plan/plan variation should be used.

This resolution must be based on:

  - product mapping
  - billing period
  - extension-managed subscription mapping data

It must not:
  - guess based on UI labels
  - guess based on invoice title text alone
  - create subscriptions without explicit mapping confidence

--------------------------------------------------

7. Subscription Creation Rule

Subscription creation must occur only after:

  - customer is ready
  - payment source is ready
  - recurring plan is resolved
  - amount/currency expectations have been validated

If recurring billing later fails for an existing subscription:
  - update internal status appropriately
  - preserve identifiers needed for retry or audit

The entire lifecycle must be idempotent.

The system must protect against:

  - multiple button clicks
  - duplicate API submissions
  - repeated webhook/callback style events
  - duplicate subscription creation for same invoice/client context

The adapter must never create multiple active subscriptions unintentionally for one intended signup.

  - when subscription was created
  - which invoice triggered creation
  - which client owns it
  - which Square subscription/customer IDs were assigned
  - what current status is

This data must remain available for admin review and troubleshooting.

15. Strict Prohibitions

The lifecycle must NOT:

  - create subscription before payment prerequisites are valid
  - create multiple subscriptions for same signup path
  - treat every recurring invoice as first-time signup
  - depend on frontend state after token submission
  - depend on banklink redirects or PayPal-style callbacks

--------------------------------------------------

16. Final Responsibility

This layer is strictly:

  → "Create and maintain the recurring billing lifecycle safely"

  → "Perform frontend tokenization"
  → "Replace transaction execution"

When a subscription is canceled:

  canceled

  pending
      recurring setup has started but is not yet fully active

10. Strict Prohibitions

  - automatically create a replacement subscription
  - erase historical recurring identifiers
  - retroactively modify already settled invoice states
  - assume cancellation affects all subscriptions for a client unless explicitly linked

--------------------------------------------------

This part of the lifecycle is strictly:

NOT:

  → "Rebuild billing automatically"
  → "Guess future customer intent"

Implementation Specification — Recovery Actions, Retry Boundaries, and Admin Visibility Rules

This section defines what the rebuilt Square payment adapter is allowed to retry, what it must never retry automatically, and how recovery actions must be bounded.

--------------------------------------------------

1. Purpose

Recovery action rules exist to prevent failure handling from creating more damage than the original problem.

The system must distinguish between:

  - safe retry opportunities
  - unsafe retry situations
  - manual recovery requirements

The adapter must never retry blindly.

--------------------------------------------------

2. Core Retry Principle

A retry is allowed only when all of the following are true:

  - the previous attempt did NOT complete successfully
  - retrying will NOT create duplicate charges or duplicate subscriptions
  - enough identifiers/state exist to safely determine what happened
  - the failure type is known to be retryable

If any of these conditions are unclear:
  → do NOT auto-retry
  → require controlled/manual handling instead

--------------------------------------------------

3. Safe Retry Categories

The implementation may support retry behavior for failures such as:

  A. temporary network/API transport issues
  B. temporary local service interruption
  C. temporary persistence/update failure after a clearly non-final step
  D. explicitly retryable pending states defined by business rules

These must only be retried if duplicate protection remains intact.

--------------------------------------------------

4. Unsafe Retry Categories

The implementation must NOT automatically retry when the prior outcome is uncertain in ways that could cause duplication.

Examples:

  - payment may already have succeeded but local confirmation failed
  - subscription may already have been created but persistence failed
  - external state cannot yet be trusted or confirmed
  - transaction status is ambiguous

In these situations:
  → preserve context
  → stop automatic flow
  → require controlled reconciliation

--------------------------------------------------

5. Payment Retry Rule

If a payment attempt fails before confirmed success:
  - the invoice must remain unpaid
  - the transaction should remain in a non-processed state
  - retry may be allowed only if idempotency protection can guarantee no duplicate charge

If a payment may already have succeeded:
  - do NOT auto-charge again
  - transition into reconciliation/review path

--------------------------------------------------

6. Subscription Retry Rule

If subscription creation fails:
  - do NOT immediately attempt to create a second subscription blindly
  - retry is allowed only when the system can prove that no active subscription was created for the same context

If that cannot be proven:
  - preserve identifiers and state
  - require sync/reconciliation before retry

--------------------------------------------------

7. Retry Limit Rule

The system must define a retry boundary.

It must not:
  - loop indefinitely
  - continuously retry in the same request lifecycle
  - hammer Square API
  - repeat user-triggered retries without resetting state responsibly

Retry policy must be explicit and conservative.

--------------------------------------------------

8. Manual Recovery Rule

When retry is unsafe or ambiguous, the system must preserve enough information for manual recovery.

Manual recovery context should make it possible to determine:

  - what step failed
  - whether payment may already have been created
  - whether subscription may already have been created
  - what invoice/client/gateway context was involved

--------------------------------------------------

9. Strict Prohibitions

Recovery logic must NOT:

  - retry blindly after unknown partial success
  - create duplicate charges
  - create duplicate recurring subscriptions
  - erase failed transaction context before analysis
  - convert ambiguous failures into fake success

--------------------------------------------------

10. Final Responsibility

This part of the lifecycle is strictly:

  → "Define when retry is safe and when recovery must stop"

NOT:

  → "Keep retrying until something works"
  → "Guess whether external actions already succeeded"

This section defines how the rebuilt Square payment adapter must treat invoice state so that billing remains correct and stable.

  - partial or ambiguous backend failures
  - retries and duplicate requests

Frontend code must never:

  - alter invoice state directly

3. Required Invoice States

The implementation must respect meaningful invoice lifecycle states such as:

  unpaid
      invoice exists but no successful settlement has occurred

  approved

  refunded / adjusted / other platform states
      handled only when explicitly supported by adapter design

The adapter must not invent conflicting invoice concepts that bypass FOSSBilling state management.

--------------------------------------------------

4. Initial Payment Rule

For a normal successful payment:

  - load the invoice
  - validate invoice amount against backend-derived payment amount
  - approve invoice if required by FOSSBilling flow

  - mark invoice paid only after transaction success is confirmed

If any required step fails:

5. Failure Rule

If payment execution fails:

  - invoice remains unpaid
  - invoice must not be approved purely because payment was attempted
  - user receives controlled failure response
  - transaction state must reflect non-success

If state becomes ambiguous:
  - prefer leaving invoice unpaid/pending review rather than falsely paid

--------------------------------------------------

6. Subscription Initial Invoice Rule

If the invoice is related to first-time recurring signup:

  - the implementation must explicitly define whether payment success alone is enough to mark invoice paid

The rebuild must not accidentally mark the invoice fully settled if the intended combined payment/subscription lifecycle is incomplete and the design requires both parts.

--------------------------------------------------

7. Renewal Invoice Rule

Renewal invoices must be treated as their own invoice events.

The system must support:

  - resolving the correct invoice being paid
  - ensuring recurring billing does not pay the wrong invoice
  - keeping invoice linkage truthful for recurring charges

Renewal invoices must not be confused with the original signup invoice.

--------------------------------------------------

8. Safety Requirement

Invoice updates must be:

  - idempotent where possible
  - resistant to duplicate requests
  - resistant to partial retry issues
  - traceable to the transaction that caused the change

--------------------------------------------------

9. Strict Prohibitions

Invoice state handling must NOT:

  - mark invoice paid before confirmed backend success
  - trust frontend payment claims
  - update invoice state from browser-only results
  - confuse one-time and recurring invoice handling
  - silently override failed states as success

--------------------------------------------------

10. Final Responsibility

This part of the lifecycle is strictly:

  → "Define when invoice state may safely change"

NOT:

  → "Control frontend behavior"
  → "Perform Square tokenization"


Implementation Specification — Invoice & Transaction State Rules

This section defines how the rebuilt Square payment adapter must manage transaction records so that payment history remains correct, auditable, and synchronized with invoice state.

--------------------------------------------------

1. Purpose

Transaction state rules exist to ensure that every payment attempt has an accurate backend record.

The rebuilt system must preserve transaction truth under:

  - successful one-time payments
  - successful recurring signup payments
  - failed payment attempts
  - duplicate submissions
  - ambiguous or partially completed external outcomes

--------------------------------------------------

2. Core Transaction Principle

A transaction record must reflect what is actually known about the payment attempt.

It must never be used to imply success unless backend confirmation exists.

Transaction state must be controlled only by backend processing.

Frontend must never:
  - mark a transaction processed
  - fabricate transaction IDs
  - alter transaction status directly

--------------------------------------------------

3. Required Transaction Outcomes

The implementation must support meaningful transaction states such as:

  received / pending
      request accepted but not yet confirmed complete

  processed
      verified successful payment outcome

  failed
      verified unsuccessful outcome

  ambiguous / needs review
      optional state if the design supports uncertain external results

These states must be explicit and not overloaded with conflicting meanings.

--------------------------------------------------

4. Creation Rule

A transaction record must exist or be resolved before payment finalization is attempted.

The adapter must be able to associate the current payment attempt with:

  - invoice_id
  - gateway_id
  - client context if needed
  - external Square identifiers after success

The transaction record must never be detached from the invoice it belongs to.

--------------------------------------------------

5. Success Update Rule

After verified successful payment execution, the transaction must be updated with:

  - status = processed

  - transaction status text from Square if relevant
  - amount

  - updated timestamp

--------------------------------------------------

6. Failure Update Rule

If payment execution fails clearly:

  - transaction must not be marked processed
  - failure state or equivalent must be recorded
  - useful sanitized error context should remain available
  - invoice must remain consistent with failure

A failed transaction must remain reviewable for retry or diagnostics.

--------------------------------------------------

7. Duplicate Protection Rule

The adapter must protect against duplicate transaction processing caused by:

  - double-clicking pay button
  - repeated frontend requests
  - network retries
  - duplicate backend execution

This must be enforced using:
  - idempotency keys for Square
  - careful transaction state handling locally

The system must never create multiple processed records for the same logical charge.

--------------------------------------------------

8. External Identifier Rule

When Square returns an external payment identifier, that identifier must be preserved in the transaction record.

This allows:

  - reconciliation
  - audit
  - duplicate detection
  - support investigation

If the external identifier is missing when success is expected:
  → treat the situation as unsafe or incomplete rather than silently successful

--------------------------------------------------

9. Separation from Invoice Rule

Transaction state and invoice state are related but not identical.

The system must not assume:

  processed transaction
      always means invoice is already finalized

or

  unpaid invoice
      means no transaction attempt exists

Both records must be updated deliberately and in the correct order.

--------------------------------------------------

10. Ambiguity Rule

If Square response is incomplete or the local update sequence is interrupted:

  - transaction must not be falsely marked as success
  - enough context must remain to reconcile later
  - duplicate charging must be avoided

When uncertain:
  → prefer conservative non-success transaction state

--------------------------------------------------

11. Auditability Rule

Each transaction should preserve enough safe information to answer:

  - which invoice was affected
  - which payment attempt occurred
  - whether Square returned success or failure
  - what external ID was assigned
  - when the state changed

This is required for support and recovery.

--------------------------------------------------

12. Strict Prohibitions

Transaction handling must NOT:

  - mark transaction processed before verified payment success
  - overwrite historical successful transaction data incorrectly
  - discard useful identifiers on failure
  - rely on frontend claims of success
  - allow duplicate processed outcomes for one intended charge

--------------------------------------------------

13. Final Responsibility

This part of the lifecycle is strictly:

  → "Define how payment attempt records are created, updated, and protected"

NOT:

  → "Render UI"
  → "Perform tokenization"
  → "Control page flow"

Error Handling Principles

This section defines how the rebuilt Square payment adapter must handle errors so that failures never break invoice rendering, never corrupt billing state, and remain diagnosable.

1. Purpose

Error handling rules exist to ensure that all failures are:

  - safe
  - controlled
  - visible
  - non-destructive
  - recoverable when appropriate

2. Core Error Principle

Every error must be handled at the lowest reasonable layer without allowing uncontrolled propagation to:

  - Twig rendering
  - invoice views
  - checkout page rendering
  - other unrelated FOSSBilling pages

If a payment component fails, the surrounding application must remain usable.

The getHtml() method must NEVER throw fatal exceptions.

  - render a controlled payment-unavailable message if needed
  - still return valid HTML string
  - avoid breaking invoice page rendering

This rule is absolute.

--------------------------------------------------

4. Backend Safety Rule

Backend payment processing methods may throw controlled exceptions, but only where the calling flow is expected to handle them safely.

These exceptions must:

  - represent a known failure state
  - avoid leaking sensitive details
  - avoid corrupting invoice or transaction state

Uncontrolled fatal errors are prohibited.

--------------------------------------------------

5. Frontend Safety Rule

Frontend JS must catch all async and runtime errors from:

  - Square SDK initialization
  - card attachment
  - tokenization
  - API request submission
  - API response parsing

Frontend must:

  - show user-friendly message
  - re-enable Pay button where appropriate
  - never rely on browser default behavior after failure

--------------------------------------------------

6. Failure Containment Rule

If any one subsystem fails, failure must remain contained.

Examples:

  frontend tokenization failure
      must not change backend state

  backend payment failure
      must not break invoice page rendering

  subscription creation failure
      must not falsely mark recurring state active

  logging failure
      must not become business logic failure

--------------------------------------------------

7. User Message Rule

User-facing errors must be:

  - clear
  - short
  - non-technical
  - safe

Examples of acceptable style:

  "Payment failed. Please try again."
  "Payment system is temporarily unavailable."
  "Unable to process card details."

User messages must never expose:
  - access tokens
  - internal SQL
  - stack traces
  - raw API payloads

8. Developer/Admin Message Rule

Developer/admin-visible errors may be more specific, but still must remain sanitized.

They may include:
  - step that failed
  - invoice or transaction context
  - safe external identifiers
  - safe error summaries

They must not include:
  - raw card data
  - secret keys
  - full authentication headers
  - unsafe personal payment data

--------------------------------------------------

9. Default Failure Behavior

When uncertain, the system must fail conservatively.

That means:

  - invoice stays unpaid
  - transaction stays non-processed
  - subscription stays non-active
  - checkout remains on page with error shown

The rebuild must prefer safe non-success over false success.

--------------------------------------------------

10. Strict Prohibitions

Error handling must NOT:

  - break unrelated pages
  - convert unknown errors into silent success
  - expose secrets to end users
  - allow exceptions in getHtml() to escape
  - mark billing objects successful after failure

--------------------------------------------------

11. Final Responsibility

This part of the lifecycle is strictly:

  → "Ensure all failures are contained, safe, and understandable"

NOT:

  → "Hide all errors"
  → "Pretend failures succeeded"

Safe Failure Outcomes & Rollback Rules

Implementation Specification — Error Handling, Logging, and Safe Failure Rules

This section defines what the rebuilt Square payment adapter must do when something fails after partial work has already happened.

--------------------------------------------------

1. Purpose

Safe failure outcome rules exist to ensure that partial failures do not leave the system in a misleading, broken, or contradictory state.

The rebuilt system must always prefer:

  - consistency
  - auditability
  - non-destructive failure
  - conservative state transitions

--------------------------------------------------

2. Core Principle

If the system cannot prove that a business action completed successfully, it must not record that action as fully successful.

Examples:

  payment may have failed
      → do not mark invoice paid

  subscription may not have been created
      → do not mark recurring record active

  local DB update may have failed
      → do not assume persistence succeeded

--------------------------------------------------

3. Rollback Objective

Rollback in this context does NOT require database-level transaction magic everywhere.

Instead, it means the system must:

  - avoid advancing local state too early
  - avoid false success markers
  - preserve enough information to reconcile partial failures later
  - stop safely at the point of failure

--------------------------------------------------

4. Safe Failure Order Rule

The implementation must order operations so that irreversible local changes happen only after required external success is known.

Example principle:

  validate first
  → call Square
  → verify result
  → update transaction
  → update invoice
  → update subscription state

This reduces the amount of rollback needed.

--------------------------------------------------

5. Payment Failure Outcome Rule

If the Square payment request fails before confirmed success:

  - invoice remains unpaid
  - transaction remains non-processed
  - subscription remains non-active
  - user receives controlled error
  - logs preserve safe failure details

The system must not create fake success from a failed charge attempt.

--------------------------------------------------

6. Partial Success Ambiguity Rule

If Square may have accepted an action, but the local system failed before finalizing state:

  - do not immediately retry blindly
  - do not mark full success unless verified
  - preserve external identifiers if available
  - move the situation into review/reconciliation-safe handling

This is especially important for:
  - payment creation
  - subscription creation
  - stored payment source creation

--------------------------------------------------

7. Local Update Failure Rule

If external Square success is confirmed but a local DB or service update fails:

  - preserve the external identifiers
  - do not discard the successful external result
  - stop automatic progression
  - surface a controlled internal failure requiring reconciliation

The system must not hide this situation.

--------------------------------------------------

8. Invoice Rollback Rule

If invoice finalization fails after payment success is known:

  - do not erase the transaction outcome
  - do not pretend the payment never happened
  - preserve enough context to safely apply the invoice update later
  - avoid charging again during recovery unless explicitly proven safe

--------------------------------------------------

9. Subscription Rollback Rule

If recurring subscription creation fails:

  - recurring record must not be marked active
  - invoice/payment state must follow explicit business rules
  - local mapping/state must remain consistent with what is truly known
  - duplicate subscription creation must be avoided during retry

If subscription creation may already have succeeded externally:
  - preserve that identifier if known
  - do not create a second one blindly

--------------------------------------------------

10. Frontend Failure Outcome Rule

If frontend fails before backend request:
  - no backend state changes occur
  - button is re-enabled
  - user stays on payment page
  - user sees a clear non-technical error

Frontend failures must never create backend side effects by accident.

--------------------------------------------------

11. Retry Boundary Rule

A safe failure outcome must leave the system in one of these states:

  A. clearly retryable
  B. clearly non-retryable
  C. clearly needs reconciliation/manual review

The system must not leave failures in an undefined middle state.

--------------------------------------------------

12. Recovery Information Rule

After a safe failure, the system must still be able to determine:

  - what step failed
  - whether payment might already exist externally
  - whether subscription might already exist externally
  - what identifiers were assigned
  - whether retry is dangerous

This is required for support and recovery design.

--------------------------------------------------

13. Strict Prohibitions

Safe failure handling must NOT:

  - mark invoice paid on uncertain result
  - mark transaction processed without verified payment success
  - mark subscription active without verified creation success
  - retry ambiguous operations automatically
  - discard external identifiers needed for reconciliation

--------------------------------------------------

14. Final Responsibility

This part of the lifecycle is strictly:

  → "Define the safe final state after failure and how partial work is preserved"

NOT:

  → "Pretend failure is success"
  → "Retry everything automatically"

Non-Negotiable Guardrails / Anti-Regression Checklist

Implementation Specification — Non-Negotiable Guardrails / Anti-Regression Checklist

This section defines the hard rules that must never be violated during the rebuild, no matter how convenient a shortcut may seem during implementation.

These rules exist specifically to prevent regression into the exact failure patterns already experienced.

--------------------------------------------------

1. Purpose

The anti-regression checklist exists to protect the rebuilt project from repeating known architectural mistakes.

It is the “do not cross this line” section of the project prompt.

Any implementation that violates these guardrails must be treated as incorrect, even if it appears to work temporarily.

--------------------------------------------------

2. Absolute Runtime Guardrails

The rebuilt Square payment adapter must NEVER:

  - post payment submissions to /invoice/banklink/
  - rely on banklink POST processing for Square
  - copy PayPal-style redirect gateway behavior
  - use nested forms inside the payment UI
  - require modification of FOSSBilling core templates for normal operation
  - break invoice rendering when enabled
  - throw fatal exceptions from getHtml()
  - cause unrelated invoice pages to fail

If any of the above occur:
  → implementation is invalid

--------------------------------------------------

3. Frontend Guardrails

The frontend payment logic must NEVER:

  - call form.submit()
  - trigger default checkout/banklink navigation
  - depend on the outer checkout form structure
  - depend on undocumented FOSSBilling JS helpers
  - depend on jQuery-only behavior
  - access raw card data
  - expose secrets in browser code
  - manipulate invoice/payment state directly

If any of the above occur:
  → frontend design is invalid

--------------------------------------------------

4. Backend Guardrails

The backend adapter must NEVER:

  - trust frontend amount values
  - mark invoices paid before verified backend payment success
  - mark transactions processed before verified Square success
  - mark subscriptions active before verified creation success
  - retry ambiguous operations blindly
  - create duplicate charges
  - create duplicate subscriptions
  - expose sensitive configuration in errors/logs
  - let payment failures break Twig rendering

If any of the above occur:
  → backend design is invalid

--------------------------------------------------

5. Extension Guardrails

The Squaremanager extension must NEVER:

  - own payment execution
  - tokenize cards
  - submit payments from browser
  - mark invoices paid
  - replace the gateway adapter as runtime payment engine
  - inject fragile checkout behavior into unrelated pages
  - create hidden coupling that makes one-time payments depend on admin mapping data
  - destabilize invoice rendering when enabled/disabled

If any of the above occur:
  → extension design is invalid

--------------------------------------------------

6. Rendering Guardrails

The system must ALWAYS preserve these guarantees:

  - invoice pages render
  - invoice pagination renders
  - payment page renders
  - enabling Square does not break unrelated invoice views
  - misconfiguration degrades safely
  - getHtml() always returns valid HTML string

If enabling Square can still break invoice browsing:
  → rebuild is not acceptable

--------------------------------------------------

7. Security Guardrails

The rebuilt project must NEVER:

  - handle raw card data in PHP
  - store raw card data locally
  - expose Square access token to frontend
  - log raw card data
  - log unmasked secrets
  - treat source token as a permanent reusable identifier unless explicitly converted through backend flow
  - bypass validation of invoice_id, gateway_id, or tokenized input

If any of the above occur:
  → implementation is unsafe

--------------------------------------------------

8. Subscription Guardrails

Recurring billing logic must NEVER:

  - run for all invoices by default
  - treat every recurring invoice as a first-time signup
  - guess plan mappings from labels alone
  - create recurring subscriptions without explicit verified mapping
  - silently reactivate canceled subscriptions
  - mark recurring state active on partial failure
  - ignore mismatch between Square and FOSSBilling state

If any of the above occur:
  → subscription lifecycle design is invalid

--------------------------------------------------

9. Recovery Guardrails

Failure recovery must NEVER:

  - hide ambiguous outcomes as success
  - auto-retry potentially successful charges
  - auto-retry potentially successful subscription creations
  - erase identifiers needed for reconciliation
  - leave support with no actionable context
  - convert uncertainty into false paid/active state

If any of the above occur:
  → recovery logic is invalid

--------------------------------------------------

10. Deployment Guardrails

Deployment/install logic must NEVER:

  - require manual edits to unrelated core files for normal use
  - depend on fragile external JS file paths unless explicitly justified
  - patch core templates as a normal requirement
  - break when run more than once
  - make adapter runtime dependent on admin UI availability
  - leave the system unstable after disable/remove

If any of the above occur:
  → deployment design is invalid

--------------------------------------------------

11. Final Anti-Regression Rule

The rebuilt project must always preserve this architecture:

  Adapter:
    runtime payment engine

  Inline JS:
    tokenization + request handoff only

  Extension:
    admin/configuration/deployment support only

If any layer begins taking over responsibilities from another layer in uncontrolled ways:
  → stop and redesign before continuing

--------------------------------------------------

12. Final Responsibility

This section is strictly:

  → "Define the hard non-negotiable boundaries that prevent regression"

NOT:

  → "Allow temporary hacks because they seem faster"
